# Glossary

Alphabetical reference for Callimachus concepts and terminology.
For a narrative introduction to these terms in context, see the [Vocabulary Primer](vocabulary-primer.md).

---

**Abstract kind** — A universal vocabulary term that maps across corpus types. Where a book corpus has `character` entities and a code corpus has `class` entities, both map to the abstract kind `person` or `component`. Enables cross-corpus queries without flattening adapter-specific precision. Populated at entity insert time via the [kind taxonomy](#kind-taxonomy). See also: [Primer §Stage 6](vocabulary-primer.md#stage-6--ontology-local-and-global).

**Adapter** — A corpus-type-specific plugin that implements the `SourceAdapter` trait. An adapter knows how to chunk a source, extract structure, identify entities, summarize content, and declare its chunk kind rollup hierarchy. Current adapters: `book`, `code`, `wiki`. The adapter is where [local ontology](#local-ontology) lives. See also: [Primer §Stage 3](vocabulary-primer.md#stage-3--chunks).

**Aliases pass** — Pipeline pass that resolves entity name variants within a corpus. The adapter's `resolve_aliases` method receives all extracted entities and returns merge instructions — e.g., `Eisenhorn` and `Gregor Eisenhorn` are the same entity. Runs after the [semantic pass](#semantic-pass), before [summarize](#summarize-pass). See also: [pass](#pass), [corrections model](#corrections-model).

**Chunk** — The atomic unit of indexed content. A corpus is divided into chunks during the chunk pass; each chunk has a [kind](#chunk-kind), a location URI, a parent path (for hierarchy), and raw content. Chunk IDs are content-addressed hashes, making re-indexing idempotent. See also: [Primer §Stage 3](vocabulary-primer.md#stage-3--chunks).

**Chunk kind** — A label describing what a chunk represents within its corpus type. Book chunk kinds: `scene`, `chapter`. Code chunk kinds: `function`, `class`, `file`, `module`, `interface`. Wiki chunk kinds: `section`, `page`. Chunk kinds are adapter-owned ([local ontology](#local-ontology)) and define the [rollup](#rollup) hierarchy for the summarize pass. Contrast with [entity kind](#entity-kind).

**Collection** — A named grouping of corpora that enables cross-corpus queries. A collection can span corpus types — a wiki and a codebase, or all three books in a trilogy. The primary mechanism for cross-corpus analysis. See also: [Primer §Stage 8](vocabulary-primer.md#stage-8--putting-it-together).

**Contract** — A structured representation of a code entity's obligations and assumptions: preconditions, postconditions, invariants, error conditions. Generated by the contract pass from the entity's source code. Analogous to a formal specification. Most meaningful for code; book/wiki adapters inherit a default no-op implementation. See also: [purpose](#purpose), [Primer §Stage 5](vocabulary-primer.md#stage-5--the-pipeline-and-its-order).

**Corpus** *(pl. corpora)* — A single indexed source: one codebase, one book, one wiki. The fundamental unit of organization. Each corpus has a kind (book/code/wiki), a source path, a status, and a [pipeline version](#pipeline-version). All corpora share one [pinakes](#pinakes), separated by `corpus_id`. See also: [Primer §Stage 2](vocabulary-primer.md#stage-2--corpus-and-corpora).

**Corrections model** — A mechanism for applying semantic fixes to an index without re-indexing. The individual records are called [scholia](#scholion); they are stored in the `scholia` table and applied at query time. Designed for fixes that don't warrant a full pipeline re-run — e.g., merging `Bequin` and `Alizebeth Bequin` after the [aliases pass](#aliases-pass) missed it. Apply via `calli scholion apply` (`calli correct` is a deprecated alias). See also: [scholion](#scholion), [Primer §Stage 7](vocabulary-primer.md#stage-7--corrections-and-evolution).

**Diegesis** — Greek for "narrative exposition". The multi-paragraph explanation produced by `explain_component` / `calli inspect diegesis`. Assembled from pre-indexed purposes, summaries, and block descriptions via BFS over `calls` edges — zero LLM calls at query time. See also: [Primer §Stage 5](vocabulary-primer.md#stage-5--the-pipeline-and-its-order).

**Edge** — A directed relationship between two entities. Carries a label (e.g., `appears_in`, `calls`, `depends_on`) and a weight. Edges populate the [entity graph](#entity-graph). See also: [Primer §Stage 4](vocabulary-primer.md#stage-4--entities-and-edges).

**Embedding** — A high-dimensional vector representation of a chunk or entity, used for semantic similarity search. Generated by the embed pass (opt-in). Requires a separate embedding provider.

**Entity** — A named concept extracted from a corpus: a character, location, function, class, concept, faction, etc. Entities have a canonical name, a [kind](#entity-kind), a description, and optionally a [summary](#summary), [purpose](#purpose), [contract](#contract), and [abstract kind](#abstract-kind). Entities are the primary query target. See also: [Primer §Stage 4](vocabulary-primer.md#stage-4--entities-and-edges).

**Entity graph** — The network of entities and edges extracted from a corpus or collection. Not a formal graph data structure but a relational representation queryable via the storage layer and MCP tools. See also: [Primer §Stage 4](vocabulary-primer.md#stage-4--entities-and-edges).

**Entity kind** — A label describing what an entity represents within its corpus type. Book entity kinds: `character`, `faction`, `location`, `concept`. Code entity kinds: `function`, `class`, `module`, `interface`. Entity kinds are adapter-owned ([local ontology](#local-ontology)) and map to [abstract kinds](#abstract-kind) in the [kind taxonomy](#kind-taxonomy). Contrast with [chunk kind](#chunk-kind).

**Epistemology** — The branch of philosophy concerned with how knowledge is acquired and justified. In Callimachus, the [pipeline](#pipeline) passes are ordered epistemologically: each pass establishes preconditions for the next. You cannot know what something *means* ([semantic pass](#semantic-pass)) until you know it *exists* ([chunk pass](#pass)); you cannot describe *intent* ([purpose pass](#purpose)) until you have established *meaning*. The ordering is not arbitrary — it encodes a claim about what must be true before something else can be known. Contrast with [ontology](#ontology). See also: [Primer §Stage 5](vocabulary-primer.md#stage-5--the-pipeline-and-its-order).

**Global ontology** — The shared vocabulary that spans all corpus types. Currently: the [abstract kind taxonomy](#abstract-kind) plus universal entity fields (purpose, contract, abstract_kind). Global ontology enables cross-kind analysis. Contrast with [local ontology](#local-ontology). See also: [Primer §Stage 6](vocabulary-primer.md#stage-6--ontology-local-and-global).

**Kind taxonomy** — The mapping table from concrete [entity kinds](#entity-kind) to [abstract kinds](#abstract-kind), keyed by `(concrete_kind, corpus_kind)`. Seeds the `abstract_kind` field on entities at insert time. Lives in the `kind_taxonomy` database table.

**Local ontology** — The vocabulary specific to one corpus type: [chunk kinds](#chunk-kind) (`scene`, `chapter`), [entity kinds](#entity-kind) (`character`, `faction`), and [rollup](#rollup) hierarchies. Owned by the [adapter](#adapter), not universal. Contrast with [global ontology](#global-ontology). See also: [Primer §Stage 6](vocabulary-primer.md#stage-6--ontology-local-and-global).

**Ontology** — The branch of philosophy (and in computing, a formal model) concerned with what kinds of things exist and how they relate. In Callimachus, ontology appears at two levels: [local](#local-ontology) (adapter-specific chunk and entity kinds) and [global](#global-ontology) (abstract kind taxonomy). Distinct from [epistemology](#epistemology): epistemology is about *how* knowledge is acquired; ontology is about *what kinds of things* the knowledge is about. See also: [Primer §Stage 6](vocabulary-primer.md#stage-6--ontology-local-and-global).

**Pass** — A single stage in the indexing [pipeline](#pipeline). Each pass operates on a corpus, processes a set of items (chunks or entities), and records a [run log](#run-log) entry. Passes are idempotent by default: already-processed items are skipped unless `--full` is specified. Standard sequence: chunk → structure → semantic → aliases → summarize → purpose → contract. Theme and embed are opt-in. See also: [Primer §Stage 5](vocabulary-primer.md#stage-5--the-pipeline-and-its-order).

**Pinakes** — Greek "tablets / lists". The catalogue Callimachus of Cyrene compiled at the Library of Alexandria. In this project: the SQLite file (`.pinakes` extension) that holds every corpus, entity, edge, summary, purpose, contract, and [scholion](#scholion). One pinakes per workspace; many corpora per pinakes. Specified via the `--pinakes` flag or `CALLIMACHUS_PINAKES` env var. See also: [Primer §Stage 2](vocabulary-primer.md#stage-2--corpus-and-corpora).

**Pipeline** — The ordered sequence of [passes](#pass) that transforms a raw source into a fully indexed corpus. The pipeline is ordered [epistemologically](#epistemology): earlier passes establish preconditions for later ones. The pipeline is corpus-type-agnostic — [adapters](#adapter) mediate all type-specific behavior. See also: [Primer §Stage 5](vocabulary-primer.md#stage-5--the-pipeline-and-its-order).

**Pipeline version** — An integer tracking the schema generation a corpus was indexed under. Incremented when new passes or schema changes are introduced. `calli status` shows each corpus's version vs. current. `calli upgrade` brings lagging corpora forward without re-indexing from source. See also: [upgrade](#upgrade), [Primer §Stage 7](vocabulary-primer.md#stage-7--corrections-and-evolution).

**Purpose** — A structured description of why an entity exists: its role, its intended use, its place in the larger system. Generated by the purpose pass. Distinct from [summary](#summary) (what it does) and [contract](#contract) (what it promises). See also: [Primer §Stage 5](vocabulary-primer.md#stage-5--the-pipeline-and-its-order).

**Rollup** — The [summarize pass](#summarize-pass)'s process of aggregating lower-level summaries into higher-level ones. Scene summaries are rolled up into chapter summaries; chapter summaries into a corpus summary. The rollup hierarchy is declared by the adapter via `summary_levels()`. See also: [chunk kind](#chunk-kind), [Primer §Stage 5](vocabulary-primer.md#stage-5--the-pipeline-and-its-order).

**Run log** — A record of each pass execution: pass name, start/end time, status, and a stats blob (processed/skipped/failed counts, token usage, cost). The authoritative source of truth for what has actually been indexed. Accessible via `calli inspect runs <corpus>`.

**Scholion** *(pl. Scholia)* — Greek "marginal note". A small, post-indexing correction that overrides part of the index without re-running the pipeline: entity merges, renames, alias edits, summary overrides, cross-corpus entity links. Stored in the `scholia` table of the [pinakes](#pinakes); applied at query time by the `CorrectionsEngine`. Managed via `calli scholion list` and `calli scholion apply`. See also: [corrections model](#corrections-model), [Primer §Stage 7](vocabulary-primer.md#stage-7--corrections-and-evolution).

**Semantic** — In common use, relating to *meaning* rather than *syntax*. In Callimachus, the semantic pass extracts entities and edges from chunks using an LLM — it moves from raw text (syntax) to structured meaning. The word also appears in the broader sense: the pipeline's output is a semantic representation of the source material. See also: [Primer §Stage 4](vocabulary-primer.md#stage-4--entities-and-edges).

**Semantic pass** — The pipeline pass that uses an LLM to extract [entities](#entity) and [edges](#edge) from each chunk. Calls the adapter's `extract_with_llm` method. The primary source of the [entity graph](#entity-graph). See also: [semantic](#semantic).

**Summary** — A human-readable prose description of a chunk or corpus. Generated by the [summarize pass](#summarize-pass) at multiple depth levels (scene, chapter, corpus for books; function, file, corpus for code). Stored in the `summaries` table. Distinct from [purpose](#purpose) (intent-focused) and [contract](#contract) (obligation-focused).

**Summarize pass** — The pipeline pass that generates [summaries](#summary) at each level declared by the adapter's `summary_levels()` method, rolling up from leaf chunks to corpus level. See also: [rollup](#rollup).

**Theme** — A cross-cutting conceptual pattern extracted across an entire corpus — recurring motifs, dominant concerns, structural patterns. Generated by the theme pass (opt-in). Currently only meaningful for corpora whose adapter implements `extract_themes`. See also: [Primer §Stage 5](vocabulary-primer.md#stage-5--the-pipeline-and-its-order).

**Upgrade** — The process of bringing a corpus's indexed artifacts forward to the current [pipeline version](#pipeline-version) without re-indexing from source. Implemented as `calli upgrade [corpus_id]`. Runs only the passes needed for each version step. See also: [corrections model](#corrections-model), [Primer §Stage 7](vocabulary-primer.md#stage-7--corrections-and-evolution).
